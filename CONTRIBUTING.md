# Contributing to WA Scheduler

Thanks for helping improve WA Scheduler. This repo is intentionally kept approachable—it doubles as a small **open source contribution exercise**, so filing issues, opening PRs, and reviewing others’ changes are all welcome skills to practice here.

## General workflow

1. Fork the repository ([Fork](https://github.com/ghazlabs/wa-scheduler/fork)).
2. Clone your fork and create a branch (`git checkout -b short-topic`).
3. Make your changes.
4. From the repository root, run **`make test`** and fix anything that fails before opening a PR.
5. Open a pull request against this repository.

Optional integration check: run the app locally with Docker as described in the [README](README.md) (Getting Started).

---

## Bug fixes

**Claim work:** On the GitHub issue for the bug, leave a comment that says exactly:

`I will try to fix this`

Do this before (or right after) you start coding so others know someone is on it.

- **Issue already exists:** Comment `I will try to fix this` if nobody has claimed it yet, then work on your PR.
- **No issue yet:** Open a new issue describing the bug, comment **`I will try to fix this`**, then implement and open your PR.

---

## Feature suggestions and UI/UX suggestions

1. **Open an issue first** describing the idea clearly enough that maintainers can discuss scope.
2. In that issue, mention **`@riandyn`** so a maintainer is notified.
3. **Wait for approval on the issue** before implementing.
4. After the issue is approved, implement your changes and open a PR.

---

## Questions?

Ask on the relevant GitHub issue, or open a new issue if you are unsure where to ask.
